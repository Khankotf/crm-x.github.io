# Дорожная карта CRM-X до SaaS

Документ описывает путь от текущего локального прототипа до первой продаваемой версии.

## Цель

Сделать стабильную первую версию, которую можно дать 1-3 пилотным клиентам.

Позиционирование:

> CRM-X — CRM для малого B2C-бизнеса: лиды, сделки, задачи, продукты, партии товаров, роли руководителей, дашборды и отчеты по продажам.

## Этап 1. Стабилизировать прототип

Зачем:

Сейчас большая часть логики находится в одном файле `app.js`. Это нормально для первой попытки, но дальше файл будет сложно развивать.

Что сделать:

- зафиксировать текущую рабочую версию;
- добавить документацию;
- добавить простые проверки;
- начать выносить код по смысловым блокам;
- не ломать текущую логику.

Что уже сделано:

- добавлены базовые документы проекта;
- добавлен единый слой хранения `crmStorage`, чтобы дальше заменить `localStorage` на API/базу данных через одно место;
- слой хранения вынесен в `src/storage.js`;
- подготовлена первая SQL-миграция для Supabase.
- добавлены `src/supabase-config.js`, `src/supabase-client.js` и `src/data-adapter.js` для будущего подключения к Supabase.

Что могу сделать я здесь:

- разделить код на модули;
- оставить прежнее поведение;
- добавить проверки;
- добавить инструкции запуска.

Что нужно от тебя:

- проверить в браузере, что после рефакторинга CRM выглядит и работает как раньше.

## Этап 2. Перенести данные из браузера в базу

Зачем:

Сейчас данные живут в браузере. Если клиент откроет CRM на другом устройстве, данных там не будет. Для продажи нужна база данных.

Рекомендуемый путь для MVP:

- Supabase;
- PostgreSQL;
- Supabase Auth;
- Row Level Security;
- frontend сначала можно оставить простым.

Что сделать:

- создать таблицы в базе;
- добавить `organization_id` во все бизнес-данные;
- сделать слой хранения данных;
- заменить прямую работу с `localStorage` на единый адаптер;
- оставить локальный режим для разработки.

Что могу сделать я здесь:

- спроектировать SQL-таблицы;
- написать миграции;
- подготовить адаптер данных;
- начать переводить приложение на работу через API/БД.

Что нужно от тебя:

- создать аккаунт Supabase;
- создать проект Supabase;
- дать URL проекта и публичный `publishable key`.

## Этап 3. Настоящая авторизация

Зачем:

Демо-логины подходят только для прототипа. Для клиентов нужен настоящий вход.

Что сделать:

- вход по email/паролю;
- восстановление пароля;
- приглашение сотрудников;
- роли: менеджер, руководитель, VIP/admin;
- запрет доступа к чужим данным.

Что могу сделать я здесь:

- подключить Supabase Auth;
- заменить текущий демо-вход;
- сделать экраны регистрации и входа;
- настроить ролевую модель в интерфейсе.

Что нужно от тебя:

- решить, регистрация будет открытая или только по приглашению;
- настроить домен для ссылок восстановления пароля, когда появится деплой.

## Рекомендуемый следующий технический шаг

Не начинать с платежей.

Сначала:

1. Разделить код на модули.
2. Добавить слой хранения данных.
3. Подготовить PostgreSQL/Supabase схему.
4. Перенести данные из `localStorage` в базу.
